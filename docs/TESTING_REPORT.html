2. Bugs Found

Bug #1 — Content Update Fails (title → name field mismatch) HIGH FIXED

Page
/dashboard/content — Edit Content modal
Behavior
Clicking “Save Changes” after editing content title shows error toast: “Failed to update content” with “Max retry attempts (3) exceeded”
Root Cause
Frontend updateContent() in api.ts sends { title: "..." } but backend expects { name: "..." }. The createContent() method correctly maps title → name, but updateContent() did not.
Fix
web/src/lib/api.ts — Added field mapping in updateContent() to convert title → name before sending to backend.

Bug #2 — Folders API Returns 500 Internal Server Error HIGH UNFIXED

Page
/dashboard/content (affects folder sidebar)
Behavior
Every page load of Content Library triggers multiple 500 errors from GET /api/folders?format=tree. Console shows: “[ContentPage] Failed to load folders: Error: Internal server error”
Root Cause
The folders endpoint returns 500 from the middleware. The content_folders table exists in the database (Prisma says schema is in sync), but the query may fail at runtime due to a service-level error. Requires middleware debug logging to pinpoint exact cause.
Fix
None — requires backend investigation with middleware debug logs.

Bug #3 — Analytics Shows Sample/Demo Data for Empty Account MEDIUM UNFIXED

Page
/dashboard/analytics
Behavior
Charts display what appears to be sample data (Testimonials views, FAQ Section views, Device Distribution percentages) for a brand-new account with 0 devices and 0 content. Only the “Usage Trends by Content Type” chart correctly shows empty data.
Root Cause
The analytics hooks (useChartData, useAnalyticsData) likely return mock/demo data when the backend returns empty datasets, or the backend analytics endpoints return seed data.
Fix
None — needs investigation into whether this is intentional demo data or a bug in the analytics data hooks.

Bug #4 — Settings Page Shows Hardcoded Organization Data MEDIUM FIXED

Page
/dashboard/settings
Behavior
Organization Name shows “My Organization” and Admin Email shows “admin@vizora.com” instead of the actual registered values (“Test Organization” and “testuser@vizora.test”).
Root Cause
Settings page uses useState with hardcoded defaults and never fetches from the API. No useEffect to load real data. “Save Changes” button has no onClick handler.
Fix
web/src/app/dashboard/settings/page.tsx — Added useAuth hook import and useEffect to populate organization name and email from the authenticated user context.

Bug #5 — Error Toast Exposes Internal IDs LOW UNFIXED

Page
/dashboard/content (when update fails)
Behavior
Error toast message includes raw internal content ID: “Max retry attempts (3) exceeded for edit_cmlazjnu40301b1l9i7ynkyh2_177038849397”
Root Cause
The useOptimisticState hook includes the operation key (which contains the entity ID) in its error messages.
Fix
None — cosmetic issue. Error messages should be user-friendly and not expose internal identifiers.

Bug #6 — Breadcrumb Formatting Inconsistency LOW UNFIXED

Page
/dashboard/settings/api-keys, /dashboard/settings/audit-log
Behavior
Breadcrumb shows “Api-keys” and “Audit-log” (raw URL slug with lowercase and hyphens) instead of “API Keys” and “Audit Log”.
Root Cause
Breadcrumb component uses the URL path segment directly without title-casing or mapping to display names.
Fix
None — cosmetic issue.

Bug #7 — Settings Page Buttons Have No Handlers MEDIUM UNFIXED

4. Remaining Issues Needing Human Attention

  1. Folders API 500 Error (Bug #2) Requires middleware debug logging to identify why GET /api/folders?format=tree returns 500. The Prisma schema is in sync, so the table exists, but the query fails at runtime. Check NestJS console output for the stack trace.
  2. Analytics Mock Data (Bug #3) Decide whether charts should show empty states or demo data for new accounts. If demo, add a “Sample data” label. If not, fix the data hooks to return empty arrays.
  3. Settings Page Backend Integration (Bug #7) Implement: Save Changes → PATCH /organizations/:id, Change Password → POST /auth/change-password, Export Data → GET /organizations/:id/export, Delete Account → DELETE /organizations/:id with confirmation modal.
  4. Error Message Sanitization (Bug #5) Clean up error toasts in useOptimisticState to not expose internal IDs.
  5. Breadcrumb Display Names (Bug #6) Add a mapping of URL slugs to display names in the breadcrumb component.

5. UX Recommendations

Content Type Change on Edit — The edit modal correctly shows “Content type cannot be changed after upload” — good UX pattern. Consider also allowing URL editing for URL-type content.
Schedule Form UX — The “Next 10 Occurrences” preview is excellent. Consider adding a conflict detection indicator when devices already have schedules at those times.
Empty States — All empty states are well-designed with helpful CTAs and icons. Consistent pattern across all pages.
Dark Mode — Comprehensive dark mode support across all pages. All text elements have proper dark: Tailwind classes.
Loading States — Consider adding skeleton loaders for the dashboard stats cards and content grid while API calls are in flight.
Notification Bell — The notification bell icon in the header could benefit from a badge/count indicator for unread notifications.
Mobile Responsiveness — Sidebar collapses correctly, but the schedule creation modal and content library filters may need mobile-specific layouts.
Keyboard Navigation — Playlist builder supports Ctrl+Z/Ctrl+Shift+Z for undo/redo — good. Consider adding keyboard shortcuts for common actions (N for new, E for edit, Delete for delete).

Overall Assessment

The Vizora web frontend is well-built with a polished UI. The main issues are:

  • Two high-severity bugs (content update field mapping, folders API error) that affect core CRUD operations
  • Settings page is partially implemented (UI only, no backend integration)
  • Analytics page shows misleading sample data

With the two fixes applied in this session and resolution of the remaining items listed above, the frontend would be in good shape for production use.
